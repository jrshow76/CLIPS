## 1. 문서 개요

### 1.1 목적
본 문서는 자동주식매매 프로그램 **TradePilot**의 비즈니스 요구사항, 기능/비기능 요구사항, 사용자 역할 및 권한 모델을 정의한다. 본 문서는 개발팀의 모든 역할(PM, DevLead, BackendSenior/Dev, FrontendSenior/Dev, Designer, QA, DBA)이 공유하는 기준 문서이다.

### 1.2 범위
- 한국 주식시장(KOSPI / KOSDAQ) 종목을 대상으로 한 데이터 수집, 기술적/모멘텀/AI 기반 분석, 자동매매 실행, 백테스트, 수익률 리포트 기능 일체.
- 증권사 API는 **대신증권 크레온(CREON Plus, Windows COM)** 단일 연동.
- 매매 모드는 **시뮬레이션(가상매매)** 과 **실거래(라이브)** 두 가지를 단일 시스템에서 토글 방식으로 운용.

### 1.3 용어 정의
| 용어 | 정의 |
|---|---|
| 시그널(Signal) | 전략의 조건이 충족되어 발생한 매수/매도 후보 이벤트 |
| 전략(Strategy) | 매매 조건과 지표 파라미터의 집합 |
| 시뮬레이션 모드 | 실제 주문은 발생시키지 않고 가상 잔고로 체결을 흉내내는 모드 |
| 실거래 모드 | 크레온 COM 객체를 통해 실제 주문이 증권사로 전송되는 모드 |
| MDD | Maximum Drawdown, 최대 누적 손실폭 |
| OBV / VWAP | On-Balance Volume / Volume Weighted Average Price |
| LSTM | Long Short-Term Memory, 단기 가격 예측에 사용되는 RNN 계열 모델 |
| 강제 청산 | 손절선/일일 손실 한도 도달 시 시스템이 강제로 보유 포지션을 매도하는 동작 |

## 2. AS-IS / TO-BE 분석

### 2.1 AS-IS (현재 상황)
- 개인 투자자는 HTS/MTS로 수동 매매 → 감정적 매매, 일관성 부재.
- 기술적 분석을 위해 별도 차트 도구, 종목 스크리너, 수익률 관리 엑셀 등을 분산 사용.
- 시뮬레이션 환경 부재로 새로운 전략 검증 불가.

### 2.2 TO-BE (목표 상태)
- 단일 플랫폼에서 분석 → 시그널 감지 → 자동매매 → 사후 리포트까지 일관된 워크플로우 제공.
- 시뮬레이션/실거래를 1-click 전환하여 전략을 안전하게 검증.
- AI/ML(LSTM) 기반 단기 예측을 보조 지표로 제공해 판단 정확도 향상.
- 업종/섹터 회전 분석과 자금 흐름을 시각화하여 종목 발굴 전략 제공.

### 3.3 실거래 전환 조건 (필수)
- 사용자 본인인증(휴대전화 + 이메일 OTP) 완료.
- 크레온 계좌 연동 정상 응답(`COM Object` 초기화 성공).
- 시뮬레이션 모드 누적 거래 30건 이상.
- 약관/리스크 고지 동의(`disclaimer_agreed_at` 값 존재).
- 일일 손실 한도(`daily_loss_limit_pct`) 등 리스크 정책 1회 이상 저장.

## 6. 시스템 환경 가정 (Assumptions)

- 크레온 API는 **Windows OS + 64bit Python** 환경에서만 동작한다. 따라서 매매 실행 워커는 Windows 서버에 분리 배포하며, 일반 API/웹 서버와는 메시지 큐 또는 RPC로 통신한다.
- 장 운영시간 외(주말/공휴일) 매매 기능은 비활성화되며, 분석/백테스트만 이용 가능하다.
- 시뮬레이션 체결가는 실제 시장가 기준으로 슬리피지 0.1% 가산하여 계산한다.

## 7. 제약사항 및 리스크

| 구분 | 내용 | 대응 |
|---|---|---|
| 기술 | 크레온 COM은 단일 인스턴스 제약(동일 PC에서 1세션) | 매매 워커를 사용자별 또는 그룹 단위로 분리 |
| 규정 | 자동매매 사고 시 법적 책임 | 약관/면책 동의 + 강제 청산 정책 + 감사 로그 |
| 운영 | 장중 장애 시 미체결 주문 처리 | 비상정지(Kill Switch) + 주문 상태 재동기화 배치 |
| 데이터 | 시세 누락/지연 | 백업 시세 소스(공공데이터/네이버 등) 보조 활용 |

## 8. 인수 기준 (Acceptance Criteria 요약)

- 모든 H(High) 우선순위 기능이 QA 시나리오를 100% 통과한다.
- 실거래 전환 절차의 모든 조건(약관/OTP/계좌연동/한도설정)이 가드되어야 한다.
- 비상정지(Kill Switch) 동작 시 5초 이내 전 주문이 취소되고 모드가 SIM으로 강제 전환되어야 한다.
- 시뮬레이션과 실거래의 결과 화면은 시각적으로 구분(색상/배지)되어야 한다.
