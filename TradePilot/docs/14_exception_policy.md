# TradePilot 예외 처리 및 정책 정의서

> 문서 ID: 14_EXCEPTION_POLICY
> 버전: v1.0
> 작성자: Planner
> 최종 수정일: 2026-05-12

본 문서는 TradePilot 전 영역의 예외 처리 정책, 에러 코드 체계, 외부 의존성 장애 대응 방침을 정의한다.

---

## 1. 에러 응답 표준

### 1.1 응답 본문 구조
```
{
  "success": false,
  "error": {
    "code": "E0021",
    "message": "일일 매수 한도를 초과했습니다.",
    "details": {
      "limit": 5000000,
      "attempted": 5500000
    },
    "trace_id": "8c0a2f6e-...",
    "ts": "2026-05-12T10:11:22+09:00"
  }
}
```

### 1.2 HTTP 상태코드 매핑
| HTTP | 사용 사례 |
|---|---|
| 200 | 정상 처리 |
| 201 | 자원 생성 성공 |
| 202 | 비동기 작업 큐 등록(백테스트, 학습 등) |
| 400 | 검증 실패(E0003) |
| 401 | 인증 실패(E0001) |
| 403 | 권한 부족(E0002), 약관 미동의(E0013) |
| 404 | 자원 없음 |
| 409 | 충돌(E0022, E0006 등) |
| 422 | 비즈니스 검증 실패(E0021, E0024 등) |
| 429 | Rate Limit 초과 |
| 502 | 외부 시스템 응답 오류(E0004, E0023) |
| 503 | 유지보수 / 의존성 단절 |
| 504 | 외부 시스템 타임아웃 |

## 2. 통합 에러 코드 체계

> 4자리 숫자 코드, 앞 2자리는 도메인 그룹.

| 그룹 | 코드 범위 | 도메인 |
|---|---|---|
| 00 | E0001~E0009 | 공통/시스템 |
| 01 | E0011~E0019 | 인증/모드 전환 |
| 02 | E0021~E0029 | 주문/매매 |
| 03 | E0031~E0039 | 백테스트 |
| 04 | E0041~E0049 | ML/예측 |
| 05 | E0051~E0059 | 사용자/계정 |
| 06 | E0061~E0069 | 시세/시장 데이터 |
| 07 | E0071~E0079 | 외부 시스템 |
| 08 | E0081~E0089 | 알림/외부채널 |
| 09 | E0091~E0099 | 운영/관리 |

| 코드 | HTTP | 메시지(샘플) | 설명/처리 |

## 3. 입력 검증 정책

### 3.1 공통 규칙
- 모든 입력은 서버에서 재검증한다(클라이언트 검증은 보조).
- 검증 실패 시 `details: { field: [msg, ...] }` 형태로 누적 반환.
- 한 요청에서 다중 필드 오류가 있어도 모두 반환한다.

### 3.2 주요 필드 규칙
| 필드 | 규칙 | 위반 시 |
|---|---|---|
| email | RFC 5322, ≤254자 | E0003 |
| password | 8~32자, 영+숫+특수 | E0055 |
| 종목코드 | `^[0-9]{6}$` | E0062 |
| 수량 | ≥1, 매매단위 배수 | E0026 |
| 가격 | 호가 단위 준수 | E0026 |
| 기간(from,to) | from ≤ to, 최대 5년 | E0063 |
| 한도 | ≥0, 계좌 한도 ≤ | E0003 |

## 4. 주문 실패 처리 정책

### 4.1 시뮬레이션 주문
- 가상 체결은 항상 즉시 성공(시장 종료 시 거부 E0007).
- 슬리피지 0.1% 가산, 수수료 0.015%(가상) 적용.
- 잔고 부족 시 E0024.

### 4.2 실거래 주문 (CREON)
| 케이스 | 코드 | 처리 |
|---|---|---|
| COM 객체 초기화 실패 | E0012 | 모드 LIVE 진입 자체 차단 |
| 주문 응답 코드 != 0 | E0023 | 응답 메시지 details 포함, 1회 재시도 후 실패 |
| 증거금 부족 | E0024 | 즉시 실패, 사용자 알림 |
| 호가 단위 오류 | E0026 | 자동 보정 옵션 ON 시 보정 후 재전송 |
| 상하한가 도달 | E0027 | 주문 거부, 시그널 IGNORED 처리 |
| 거래 정지 | E0028 | 해당 종목 매매 불가 처리(24시간 캐시) |
| 타임아웃 | E0072 | 최대 3회 지수 백오프(1s, 2s, 4s) |

### 4.3 부분 체결 / 미체결
- 부분 체결: 잔량은 5분간 유지 후 자동 취소(전략 옵션으로 조정 가능).
- 미체결 일괄 취소: 장 종료 5분 전 자동 트리거.

### 4.4 중복 주문 방지
- `X-Idempotency-Key` 미사용 호출은 동일 사용자 + 종목 + 사이드 + 수량 + 60초 윈도우 기준 중복 차단(E0022).
- 키 사용 시 24시간 내 동일 요청은 기존 결과 반환.

## 5. 데이터 결손 처리 정책

| 상황 | 처리 |
|---|---|
| 캔들 누락 구간 | 응답에 `null` OHLC, 클라이언트는 점선/공백 표시 |
| 실시간 시세 미수신(>3s) | 직전 시세 사용 + "지연" 배지(E0061) |
| 마스터 데이터 누락(종목명 등) | 종목코드만 표시, 야간 배치로 보강 |
| 지표 산출 데이터 부족 | 산출 가능 시점부터 반환, 앞 구간은 null |
| 백테스트 종목 일부 데이터 부족 | 해당 종목 스킵 + 결과에 `skipped_codes[]` 포함 |
| ML 미학습 종목 | E0041 + 학습 대기열 등록 |

### 5.1 마감/휴장 처리
- 휴장일에는 시세/지표 API는 직전 거래일 데이터 반환.
- 매매 관련 API는 E0007 반환.

## 6. 외부 시스템 장애 대응

### 6.1 CREON
| 상태 | 감지 방법 | 대응 |
|---|---|---|
| COM 초기화 실패 | 최초 연결 시 | LIVE 진입 차단(E0012) |
| 세션 단절 | 5초 주기 헬스체크 | 자동 재연결 최대 3회 → 실패 시 자동매매 일시중지 |
| 주문 거절 | 응답 코드 | 종류별 매핑 처리(상기 4.2) |
| 응답 지연 | 5초 초과 | E0072 + Backoff 재시도 |

### 6.2 시세 백업 소스
- 1차: 크레온 실시간 시세.
- 2차: 공공 데이터 / 네이버 페이지(스크래핑) → fallback 시 데이터에 `source: "fallback"` 표기.

### 6.3 알림 채널
| 채널 | 실패 처리 |
|---|---|
| 인앱 | 즉시 DB 저장(필수 보장) |
| 이메일 | SMTP 실패 시 큐 재시도(최대 5회, 지수 백오프) |
| 텔레그램 | 실패 시 로그 후 이메일 fallback |

### 6.4 DB / Redis / 메시지 큐
- DB 단절: 헬스체크 실패 시 메인터넌스 모드 자동 전환(E0091, 503 반환).
- Redis 단절: 캐시 없이 DB 직접 조회로 graceful degrade, 알람 발송.
- 메시지 큐 단절: 백테스트/알림 요청 거부(E0004).

## 7. 리스크 기반 자동 차단 정책

| 트리거 | 조치 | 알림 |
|---|---|---|
| 일일 손실 한도(`daily_loss_limit_pct`) 도달 | 자동매매 OFF, 신규 매수 차단, 보유 종목 청산 옵션 실행 | 이메일 + 인앱 |
| 종목별 손절선 도달 | 시장가 매도 주문 | 인앱 |
| 슬리피지 ≥ 임계값 | 해당 종목 30분 매매 중단 | 인앱 |
| 동일 종목 5분 내 5회 이상 시그널 | 30분 매매 중단(쿨다운) | 인앱 |
| 시스템 오류율 ≥ 5%(5분) | 자동매매 일시중지, 운영자 호출 | 이메일 + SMS |

## 8. 비상정지(Kill Switch) 처리 흐름
1. 사용자 또는 시스템에서 Kill Switch 트리거.
2. 활성 전략 일괄 비활성화(`active=false`).
3. 진행중 주문 전체 취소 요청 → 결과 집계.
4. 모드 강제 SIM으로 전환.
5. 감사 로그(`audit_log`) 기록(트리거 주체, 사유, 처리 건수).
6. 인앱/이메일 알림 발송.
- 일부 취소 실패 시 E0015 반환, 미처리 주문 ID 리스트 details 포함.

## 9. 로깅 / 감사

### 9.1 로그 레벨 정책
| 레벨 | 대상 |
|---|---|
| DEBUG | 개발 환경 한정, 외부 호출 본문 |
| INFO | 정상 비즈니스 이벤트 |
| WARN | 자동 복구 가능한 이슈(재시도 성공 등) |
| ERROR | 사용자 영향 있는 실패 |
| CRITICAL | 자동매매 중단, 보안 사고, DB 단절 |

### 9.2 감사 로그(audit_log) 필수 기록
- 로그인/로그아웃, 권한 변경.
- 매매 모드 전환, 한도 변경.
- 주문 생성/취소/강제 청산.
- Kill Switch, 강제 청산.
- 관리자 API 호출 일체.

### 9.3 PII / 민감정보 마스킹
- 비밀번호/토큰 절대 로깅 금지.
- 이메일은 `a***@b.com`, 휴대전화는 `010-****-1234` 마스킹.

## 10. 사용자 메시지 톤 가이드

| 상황 | 권장 메시지 |
|---|---|
| 일반 검증 오류 | "{필드명}을(를) 확인해주세요." |
| 외부 장애 | "현재 시세 서버 응답이 지연되고 있습니다. 잠시 후 다시 시도해주세요." |
| 한도 초과 | "오늘 매수 한도를 초과했습니다. 설정에서 한도를 확인하세요." |
| 보안 이슈 | "보안을 위해 다시 로그인해주세요." |
| 강제 청산 | "리스크 정책에 따라 보유 종목이 청산되었습니다. 상세 내역을 확인하세요." |

- 사용자 메시지는 책임 소재가 사용자/시스템 어디인지 명확히 한다.
- 기술 세부(스택 트레이스 등)는 사용자에게 노출하지 않는다.
