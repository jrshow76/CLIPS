## 1. 리스크 평가 기준

### 1.1 확률 (Likelihood)

| 등급 | 정의 | 수치 |
|---|---|---|
| 매우 높음 | 발생이 거의 확실 | 5 |
| 높음 | 자주 발생 가능 | 4 |
| 중간 | 발생 가능 | 3 |
| 낮음 | 드물게 발생 | 2 |
| 매우 낮음 | 발생 거의 없음 | 1 |

### 1.2 영향도 (Impact)

| 등급 | 정의 | 수치 |
|---|---|---|
| 치명적 | 프로젝트 중단·금전적 손실 | 5 |
| 큼 | 마일스톤 1주 이상 지연 | 4 |
| 보통 | 마일스톤 2~3일 지연 | 3 |
| 작음 | 일정 영향 미미, 우회 가능 | 2 |
| 매우 작음 | 영향 없음 수준 | 1 |

### 1.3 리스크 점수 = 확률 × 영향도 (최대 25)
- 1~6: 낮음 (모니터링)
- 7~14: 중간 (관리 계획 수립)
- 15~25: 높음 (즉시 대응 필요)

## 3. 긴급 중단(Kill Switch) 절차

> 실거래 사고 또는 시장 급변 상황에서 즉시 모든 매매를 중단하는 표준 절차

### 3.1 트리거 조건
- **자동 트리거**:
  - 일일 누적 손실 -3% 도달
  - 크레온 게이트웨이 헬스체크 3회 연속 실패
  - 서킷브레이커·사이드카 발동 감지
  - 종목 분당 변동률 5% 초과 (해당 종목 한정)
- **수동 트리거**:
  - 사용자의 Kill Switch 버튼 클릭
  - PM·DevLead 운영 명령

### 3.2 실행 단계
1. **즉시(0초)**: 신규 주문 큐 차단 (Order Router 차단 플래그 ON)
2. **5초 이내**: 미체결 주문 일괄 취소 요청 (`CancelAllOrders` API)
3. **10초 이내**: 매매 엔진 일시정지 상태 진입, 알림 발송
4. **30초 이내**: PM·DevLead에 SMS/메신저 알림, 운영 로그 기록
5. **사후**: 사건 보고서 작성, 원인 분석 회의 (24시간 이내)

### 3.3 재가동 절차
- DevLead 점검 → PM 승인 → 시뮬레이션 모드 1시간 검증 후 실거래 재개
