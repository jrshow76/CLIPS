# TradePilot CREON 모의투자 통합 테스트 계획서

> 문서 ID: 60_CREON_INTEGRATION_PLAN
> 버전: v1.0
> 작성자: BackendSenior
> 검토자: DevLead, QA, PM
> 최종 수정일: 2026-05-12

본 문서는 CREON Plus 모의투자(가상매매) 환경을 활용한 TradePilot 통합 테스트 전략과 단계별 게이트(Stage 0 ~ Stage 4)를 정의한다. 본 계획서는 **모의투자 단독으로 진행하는 검증**의 범위이며, 실거래(LIVE) 전환은 별도 문서 `62_go_live_checklist.md` 의 절차를 따른다.

---

## 1. 검증 목적

| 목적 | 설명 |
|---|---|
| 기능 동등성 | mock 어댑터와 실제 CREON 어댑터의 응답 포맷이 동일함을 검증 |
| 통신 안정성 | 1주일 연속 운영에서 단절/재연결/메시지 누락 0건 |
| 응답 성능 | 주문 발주 → 게이트웨이 응답까지의 지연이 목표 SLA 이내 |
| 정책 일관성 | 7단계 LIVE 전환 게이트 / Kill Switch / 한도가 실 통신 환경에서도 동일하게 동작 |
| 운영 절차 | 일일 운영 런북이 실제 환경에서 실행 가능함을 입증 |

---

## 2. 단계별 게이트 (Stage 0 ~ Stage 4)

각 Stage는 **정량 통과 기준**을 모두 충족해야 다음 단계로 진입한다.
어느 한 항목이라도 미달이면 원인 해결 후 동일 Stage 재실행.

### Stage 0 — Mock 회귀 (오프라인)
- **목표**: CREON 없이 게이트웨이 + 본체의 회귀 통과.
- **환경**: Linux/CI. `CREON_USE_MOCK=true`.
- **수행**:
  - 게이트웨이 단위 테스트 (`creon-gateway/tests/`) 100% 통과
  - 본체 통합 테스트 (`backend/tests/integration/test_creon_e2e.py`) 통과
  - QA 회귀 (`52_trading_policy_tests.md` 자동화 부분) 통과
- **통과 기준** (모두 충족):
  | 항목 | 기준 |
  |---|---|
  | 게이트웨이 단위 테스트 | 통과율 100%, 커버리지 ≥ 80% |
  | 본체 e2e 테스트 | 통과율 100% |
  | 정책 회귀 (52_tests P0) | 통과율 100% |
  | Lint / Type | mypy / ruff 0건 |

### Stage 1 — CREON 연결 검증 (Windows 호스트, 무주문)
- **목표**: Windows 호스트에서 CREON COM 연결 + 계좌 로드 + 시세 조회까지만 검증.
- **환경**: Windows 10/11 + CREON Plus 모의투자 로그인. `CREON_TRADE_ENV=SIM`, `CREON_USE_MOCK=false`.
- **수행**:
  - `start-gateway.ps1` 정상 기동
  - `GET /readyz` ok=true
  - `GET /account` 모의투자 계좌 1개 이상 반환
  - `GET /market/quote/005930` 정상 응답
  - `GET /stocks/master/005930` 종목명 "삼성전자" 반환
  - 헬스비트 30초 주기로 본체 Redis에 도착
- **통과 기준**:
  | 항목 | 기준 |
  |---|---|
  | `/readyz` ok=true | 10회 연속 성공 |
  | 시세 조회 응답 지연 | p95 < 500ms |
  | 헬스비트 누락 | 1시간 0건 |
  | 계좌 접두사 검증 | `CREON_ACCOUNT_PREFIX_SIM` 매칭 |

### Stage 2 — 모의투자 주문 검증 (소량)
- **목표**: 모의투자 환경에서 주문 발주 → 체결 → 본체 반영까지 E2E 검증.
- **환경**: Stage 1 + 모의투자 잔고 충분히 입금.
- **종목**: 변동성 낮은 대형주 3종목 우선
  - 삼성전자 `005930`
  - SK하이닉스 `000660`
  - NAVER `035420`
- **수행 시나리오** (TC-INT 케이스 참조 `61_creon_integration_cases.md`):
  1. 단순 매수 1주 (지정가) → 체결 확인
  2. 단순 매도 1주 (지정가) → 체결 확인
  3. 시장가 매수 1주 → 즉시 체결 확인
  4. 미체결 주문 취소
  5. 정정 주문 (가격 변경) — v1.1 예정 (v1.0 범위 외)
  6. 100건 연속 주문 (요청 제한 검증)
  7. Kill Switch 발동 → 5초 내 미체결 전체 취소
- **통과 기준**:
  | 항목 | 기준 |
  |---|---|
  | 주문 100건 모두 체결 | 100% (모의투자 즉시 체결 가정) |
  | 평균 체결 지연 (주문 발주 → tp:account.execution 수신) | < 800ms |
  | 95퍼센타일 체결 지연 | < 1500ms |
  | 본체 DB orders 테이블 일관성 (주문수 == 게이트웨이 발주수) | 100% |
  | 멱등성 (동일 idempotency_key 2회 발주) | 중복 발주 0건 |
  | 요청 제한 위반 | 0건 (게이트웨이 자동 페이싱) |
  | Kill Switch SLA | 5초 이내 미체결 0건 |

### Stage 3 — 모의투자 1주일 연속 운영
- **목표**: 5영업일 연속 자동매매를 모의투자 환경에서 운영하여 운영 안정성 검증.
- **환경**: Stage 2 + 등록된 전략 1개 (예: SMA 골든크로스).
- **운영 시간**: 매일 09:00 ~ 15:30 KST.
- **모니터링 지표**:
  - 일일 헬스비트 누락 건수
  - 일일 주문 발주/체결/실패 건수
  - 일일 슬리피지 평균/최대
  - 일일 PnL (모의)
- **통과 기준**:
  | 항목 | 기준 |
  |---|---|
  | 5영업일 연속 운영 | 중단 없음 |
  | CREON 단절 | 일별 0~1건 (자동 재연결 성공) |
  | 주문 실패율 | < 0.5% |
  | 평균 슬리피지 | < 0.2% (모의 환경) |
  | 일일 손익 한도 위반 | 0건 |
  | Kill Switch 동작 | 시뮬레이션 1회 발동 성공 |
  | 운영 알림 (이메일/인앱) | 100% 정상 수신 |

### Stage 4 — 실거래 전환 게이트 (LIVE Go/No-Go)
- **목표**: Stage 3 통과 후 실거래로 전환 가능한지 PM/DevLead 사인오프.
- **환경**: Stage 3 환경 + 실계좌 입금 (소액).
- **수행**: `62_go_live_checklist.md` 의 모든 항목 통과.
- **통과 기준** (`62` 문서 인용):
  - 보안 9개 항목 OK
  - 한도 5개 항목 OK
  - Kill Switch SLA 측정 OK
  - 백업 / 복구 리허설 통과
  - 모니터링 24x7 알림 채널 동작
  - 법무 / 약관 / 면책 사인오프
  - PM + DevLead + QA + 사용자 4자 사인 완료

## 3. 사용 종목군 (변동성 낮은 대형주)

| 종목명 | 코드 | 시장 | 선정 이유 |
|---|---|---|---|
| 삼성전자 | 005930 | KOSPI | 시총 1위, 호가 촘촘, 거래량 풍부 |
| SK하이닉스 | 000660 | KOSPI | 시총 상위, 변동성 적정 |
| NAVER | 035420 | KOSPI | 거래량 풍부, 정정/취소 검증 적합 |
| 카카오 | 035720 | KOSPI | 보조 종목 (확장) |
| KODEX 200 | 069500 | ETF | ETF 주문 검증 (v1.1) |

**선정 원칙**:
- 일일 거래대금 ≥ 500억
- 변동성(20일 ATR / 종가) ≤ 3%
- 호가단위가 단순 (대부분 100원)
- 거래정지/관리종목 이력 없음

**금지 종목**:
- 관리종목, 투자유의/위험 종목
- 거래량 일평균 < 10억원
- 상장폐지 임박 종목

## 4. 종료 기준 (Exit Criteria)

| 항목 | 조건 |
|---|---|
| **정상 종료** | Stage 0 ~ Stage 4 모두 통과, PM/DevLead/QA/사용자 사인오프 |
| **조기 종료 / 롤백** | 아래 중 하나 발생 시 |

### 4.1 롤백 트리거
1. **데이터 손실**: 주문-체결 불일치가 1건이라도 발생.
2. **금전 손실 위험**: 모의 환경임에도 시스템 결함으로 한도 초과 주문.
3. **연결 안정성**: Stage 3에서 일일 CREON 단절 3회 이상.
4. **응답 지연**: Stage 2 평균 체결 지연이 1000ms를 초과.
5. **Kill Switch SLA 위반**: 5초 SLA 미달성 1회.

### 4.2 롤백 절차
1. 자동매매 즉시 정지 (Kill Switch 발동).
2. 미체결 주문 일괄 취소.
3. 게이트웨이 `CREON_FORCE_MOCK=true` 로 강제 mock 전환.
4. PM/DevLead에게 사고 보고서 제출 (1시간 이내).
5. 근본 원인 분석(RCA) 후 fix → Stage 재진입.
