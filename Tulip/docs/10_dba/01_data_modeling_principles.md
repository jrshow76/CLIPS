## 2. 멀티테넌트 격리 전략

### 2.1 결정사항 (권고안)

**채택: Shared-Schema + `tenant_id` 컬럼 + PostgreSQL Row Level Security(RLS)**

| 격리 수준 | 채택 여부 | 적용 영역 |
|---|---|---|
| Database-per-Tenant | 미채택(기본) / 옵션 채택 | 대형 단일 기관(예: 광역시 통합) 별도 인스턴스 — Y2 이상 |
| Schema-per-Tenant | 미채택 | 테넌트 수 1,000 도달 시 스키마/카탈로그 부담 過大 |
| **Shared-Schema + tenant_id + RLS** | **채택 (기본)** | 전 영역 |
| Shared (no tenant_id) | 한정 적용 | 코드/표준 마스터(분류표·국가 코드 등) |

### 2.2 채택 근거

1. **확장성**: Planner 비기능 요구사항(`테넌트 1,000개·회원 1천만·서지 5천만`)을 단일 클러스터로 운영하려면 Shared-Schema가 유일하게 현실적이다. Schema-per-Tenant의 경우 1,000개 스키마 × 100+ 테이블 = 100,000+ DDL 객체로 카탈로그·백업·DDL 마이그레이션이 폭증한다.
2. **운영성**: 스키마 마이그레이션 1회 실행으로 모든 테넌트 적용. Schema-per-Tenant는 테넌트별 마이그레이션 동기화 부담이 크다.
3. **보안**: RLS로 행 단위 격리, 잘못된 쿼리도 다른 테넌트 데이터 접근 불가. JWT의 `tenant_id` 클레임을 `SET app.tenant_id = ...` 세션 변수로 전파.
4. **호환성**: 백업/복구·통계(KOLIS/KERIS/DLS 양식)·테넌트 전환·플랫폼 운영(`X-Tenant-Id` 헤더) 모두 단일 DB에서 처리 가능.

### 2.3 대형 테넌트 예외 처리

- 단일 테넌트에서 트랜잭션 10M/일, 서지 500만 이상 도달 시 별도 인스턴스(Database-per-Tenant) 분리.
- 분리 시 마이그레이션 도구로 동일 스키마·tenant_id 유지하여 코드 변경 없이 라우팅.

### 2.4 RLS 적용 정책

| 적용 대상 | 정책 |
|---|---|
| 모든 업무 테이블 (tenant_id 보유) | RLS 활성화 + `tenant_isolation` 정책 |
| 코드/표준 마스터 (tenant_id 없음) | RLS 비활성, 전 테넌트 SELECT 공유 |
| 플랫폼 운영 테이블 (tenant 자체) | 슈퍼관리자 Role만 BYPASS |

```sql
-- 예시 (loan 테이블)
ALTER TABLE tlp_cir_loan ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON tlp_cir_loan
  USING (tenant_id = current_setting('app.tenant_id')::BIGINT);
```

애플리케이션(Spring Boot) 커넥션 획득 후 첫 명령으로 `SET LOCAL app.tenant_id = :jwtTenantId;` 실행. BackendSenior와 정합 필요.

## 3. 라이브러리(분관) 키 정책 — 2계층 격리

### 3.1 계층 구조

```
tenant (조직)         — 격리 최상위. RLS 분리 대상.
  └─ library (관)     — 다관 운영 시 정책·채번·소장 단위.
       └─ location   — 서가/구역.
```

### 3.2 키 정책

- **모든 업무 테이블은 `tenant_id BIGINT NOT NULL` 보유.**
- **다관 운영에 종속되는 테이블은 `library_id BIGINT NOT NULL` 추가 보유.** 단관 운영 테넌트도 기본 library(seed) 1개를 강제 생성하여 NULL을 허용하지 않는다.
- 테넌트 전역 데이터(예: 정책 마스터·회원·서지)는 `library_id NULL` 가능하지만, **데이터 격리 단위 컬럼 자체는 NOT NULL이 원칙이며 sentinel 값(0=tenant 공통)을 사용**한다.

### 3.3 컬럼 적용 매트릭스 (요약)

| 도메인 | 테이블 예시 | tenant_id | library_id |
|---|---|---|---|
| CMN | tenant | (자기참조) | - |
| CMN | library | NOT NULL | (자기참조) |
| CMN | member | NOT NULL | NULL 가능 (테넌트 단위) |
| CAT | bibliography | NOT NULL | NULL (서지는 테넌트 공유) |
| COL | holding | NOT NULL | NOT NULL (관 단위 소장) |
| COL | copy(item) | NOT NULL | NOT NULL (등록번호 관 분기) |
| ACQ | purchase_order | NOT NULL | NOT NULL |
| CIR | loan | NOT NULL | NOT NULL (대출 발생 관) |
| ACS | access_event | NOT NULL | NOT NULL |
| FAC | seat / seat_reservation | NOT NULL | NOT NULL |

## 4. PK / FK 정책

### 4.1 PK 채택

- **기본: `BIGINT GENERATED BY DEFAULT AS IDENTITY` (`Snowflake-like` 단일 BIGINT)**
- **사유**:
  1. 인덱스 크기(8 byte)와 조인 성능에서 UUID(16 byte) 대비 우위.
  2. 테이블 파티셔닝·복제·통계에서 BIGINT 친화적.
  3. URL 노출 우려는 ULID 또는 별도 `public_id` 컬럼으로 완화.
- **예외(ULID/UUID 채택 대상)**: 외부 노출 식별자가 필요한 경우 별도 `public_id ULID UNIQUE` 컬럼 추가. (회원/서지의 외부 OPAC URL 등)
- **글로벌 시퀀스**: 모든 BIGINT PK는 단일 시퀀스 정책으로 충돌·재사용 방지. (`GENERATED BY DEFAULT AS IDENTITY` 사용 — `SERIAL` 미사용)

### 4.2 PK 명명

- 컬럼명: `id` (단순) — 모든 PK 컬럼명은 `id`로 통일.
- 제약명: `pk_{table_name}` — 예: `pk_tlp_cir_loan`.

### 4.3 FK 정책

- 컬럼명: `{참조테이블의약어}_id` — 예: `bibliography_id`, `member_id`, `library_id`.
- 제약명: `fk_{table}_{ref_table}_{ref_col}` — 예: `fk_loan_member_id`.
- `ON DELETE` 기본 정책: 업무 핵심 FK는 `RESTRICT`(또는 NO ACTION). 감사·이력 테이블은 `CASCADE` 신중 검토.
- 소프트 삭제 정책과 결합 — 실제 DELETE는 드물고 `deleted_at`으로 처리.

### 4.4 복합 유니크 제약

- `tenant_id + library_id + 비즈니스키` 형태로 자연키 유니크 제약. 예:
  - `uk_copy_accession`: `(tenant_id, library_id, accession_no)`
  - `uk_bib_isbn`: `(tenant_id, isbn)` (테넌트 내 ISBN 유일성 정책 채택 시)

## 5. 명명 규약 (Naming Convention)

### 5.1 테이블

- 패턴: `tlp_{도메인약어}_{명사_snake_case}`
- 도메인 약어: `cmn`(공통), `acq`(수서), `cat`(목록), `cir`(열람), `col`(장서), `acs`(출입), `fac`(시설)
- 이력 테이블 접미: `_history` / `_log` / `_audit`
- 코드 마스터: `tlp_cmn_code_group`, `tlp_cmn_code`
- 예시:
  - `tlp_cmn_member`
  - `tlp_cat_bibliography`
  - `tlp_cat_marc_field`
  - `tlp_col_holding`
  - `tlp_col_copy`
  - `tlp_cir_loan`
  - `tlp_acs_access_event`
  - `tlp_fac_seat_reservation`

### 5.2 컬럼

- snake_case, 모두 소문자, 약어 사용 자제 (예외: `isbn`, `marc`, `id`, `url`).
- 명사 + 명사 또는 명사 + 형용사 — 예: `accession_no`, `loan_due_date`, `is_active`.
- BOOLEAN 컬럼: `is_*` / `has_*` / `*_yn`(레거시 호환용은 미채택, `is_*` 표준).
- 날짜/시간:
  - 날짜만: `*_date` (`DATE`)
  - 시각: `*_at` (`TIMESTAMPTZ`)
  - 기간: `*_from` / `*_to`
- 상태/유형 컬럼: `*_status`, `*_type`, `*_code`.

### 5.3 인덱스

- 패턴: `ix_{table_short}_{col1}_{col2}` (일반), `uk_{table_short}_{cols}` (UNIQUE), `fx_*` (Functional/Expression), `gx_*`(GIN), `bx_*`(BRIN), `tx_*`(trgm).
- 예: `ix_loan_member_id_status`, `gx_bib_title_tsv`, `bx_access_event_event_at`.

### 5.4 시퀀스 / 트리거 / 뷰

- 시퀀스: `seq_{table}_{col}` (IDENTITY 자동 시퀀스는 PostgreSQL 자동 명명 사용).
- 트리거: `trg_{table}_{event}_{purpose}` — 예: `trg_loan_biu_audit` (Before Insert/Update Audit).
- 뷰: `vw_{도메인}_{명사}` — 예: `vw_cir_overdue_loan`.
- 파티션 자식: `{부모}_y{YYYY}` 또는 `{부모}_y{YYYY}m{MM}` — 예: `tlp_acs_access_event_y2026m05`.

## 6. 공통 컬럼 (Audit / Soft Delete / Optimistic Lock)

모든 업무 테이블은 다음 공통 컬럼을 가진다.

| 컬럼 | 타입 | NULL | 기본값 | 설명 |
|---|---|---|---|---|
| `id` | BIGINT GENERATED BY DEFAULT AS IDENTITY | NOT NULL | - | PK |
| `tenant_id` | BIGINT | NOT NULL | - | 테넌트 격리 |
| `library_id` | BIGINT | NOT NULL | - | 다관 식별 (단관은 sentinel) |
| `created_at` | TIMESTAMPTZ | NOT NULL | `now()` | 생성 시각 (UTC) |
| `created_by` | BIGINT | NULL | - | 생성자 user_id (시스템은 NULL) |
| `updated_at` | TIMESTAMPTZ | NOT NULL | `now()` | 수정 시각 (트리거 갱신) |
| `updated_by` | BIGINT | NULL | - | 수정자 user_id |
| `deleted_at` | TIMESTAMPTZ | NULL | - | 소프트 삭제 시각 |
| `version` | INT | NOT NULL | 0 | 낙관락 (`@Version`) |

### 6.1 트리거 표준

```sql
-- updated_at 자동 갱신 트리거 (모든 업무 테이블 공통)
CREATE OR REPLACE FUNCTION fn_set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at := now();
  NEW.version := COALESCE(OLD.version, 0) + 1;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### 6.2 소프트 삭제 규칙

- DELETE는 원칙적으로 `UPDATE deleted_at = now()`로 대체.
- 모든 조회 쿼리·뷰는 `deleted_at IS NULL` 조건 추가 (또는 RLS 정책에 포함).
- 물리 삭제는 다음 케이스에 한정:
  1. 개인정보 파기(회원 탈퇴 5년 경과) — 익명화 후 DELETE.
  2. 로그성 데이터 보존기간 만료 — 파티션 DROP.
  3. 테스트/스테이징 환경 데이터 클린업.

### 6.3 낙관락(Optimistic Lock)

- `version INT` 컬럼 + JPA `@Version` 매핑. (BackendSenior와 정합)
- 트리거에서 자동 증가 (위 `fn_set_updated_at()` 참고).
- 동시편집 위험이 큰 서지(`tlp_cat_bibliography`)·정책 마스터에서 특히 중요.

## 7. 코드 / 마스터 데이터 패턴

### 7.1 코드 그룹·코드값 모델

```
tlp_cmn_code_group       (1) ── (N) tlp_cmn_code   ── (N) tlp_cmn_code_i18n
- group_code (PK 자연키)              - group_code  + code (PK 자연키)
- group_name                          - sort_order
- description                         - is_active
                                      - extra (JSONB)
```

- 그룹 코드 예시: `MEMBER_TYPE`, `ITEM_TYPE`, `LOAN_STATUS`, `WITHDRAWAL_REASON`, `CURRENCY`, `LANGUAGE`, `KDC`, `DDC`, ...
- `tlp_cmn_code_i18n`: 다국어 라벨 (`locale + group_code + code`).
- 분류표(KDC/DDC/LC)는 별도 마스터 `tlp_cmn_classification` (계층 구조 — `parent_code` + `level`).

### 7.2 공유 vs 테넌트 확장 코드

- 시스템 표준 코드: `tenant_id = 0` (sentinel) — 모든 테넌트 공유.
- 테넌트 확장 코드(CMN-037): 동일 `group_code` 내 `tenant_id = <테넌트>` 행 추가 — 우선순위 적용.
- 조회 시 `WHERE tenant_id IN (0, :tenant_id) ORDER BY tenant_id DESC LIMIT 1` 패턴.

## 8. 이력 관리 패턴

### 8.1 패턴 비교

| 패턴 | 적용 도메인 | 채택 여부 |
|---|---|---|
| **History 테이블(쌍둥이)** | 회원·서지·정책·소장·등록번호 변경 | **채택(기본)** |
| **Temporal Table (System-Versioned)** | PostgreSQL 네이티브 미지원 → 미채택 | - |
| **Audit JSONB diff** | 정책 변경·권한 변경·기타 가벼운 변경 | 보조 채택 |
| **Event Sourcing** | 출입/대출 트랜잭션 (사실상 이미 이벤트 형태) | 도메인별 채택 (CIR/ACS) |

### 8.2 History 테이블 패턴

- 원본: `tlp_cat_bibliography` → 이력: `tlp_cat_bibliography_history`
- 컬럼: 원본 컬럼 + `history_id BIGINT PK`, `op_type CHAR(1)`(I/U/D), `op_at TIMESTAMPTZ`, `op_by BIGINT`, `op_reason TEXT`.
- BEFORE UPDATE/DELETE 트리거로 자동 적재.

### 8.3 Audit Log 통합 (CMN-080~083)

- 권한·로그인·개인정보 접근은 별도 `tlp_cmn_audit_log` (JSONB diff).
- 5년 보존, 1년 이상 데이터는 콜드 스토리지(파티션 분리).

### 8.4 이력 적용 매트릭스

| 테이블 | 이력 방식 |
|---|---|
| tlp_cmn_member | History 테이블 (PII 변경 추적 PIPA 의무) |
| tlp_cat_bibliography | History 테이블 (편목 이력) |
| tlp_cat_authority | History 테이블 |
| tlp_cmn_policy_rule | History 테이블 |
| tlp_col_copy | item_status_history (상태 전이 이벤트 행) |
| tlp_cir_loan | append-only(상태 컬럼 + 별도 loan_event 행) |
| tlp_acs_access_event | append-only (이미 이벤트) |
| 권한/로그인/PII접근 | tlp_cmn_audit_log (JSONB) |

## 9. 시스템 표준 (Encoding / Collation / Timezone / 시퀀스)

| 항목 | 표준 |
|---|---|
| 문자 집합 | UTF-8 (`server_encoding=UTF8`) |
| Collation | `ko-x-icu` (ICU collation, PostgreSQL 15+ 권장) |
| Timezone | DB는 `UTC`, 표시 변환은 애플리케이션에서 처리 (`Asia/Seoul`) |
| Locale | `ko_KR.UTF-8` (운영체제), DB는 ICU |
| 시퀀스 정책 | `GENERATED BY DEFAULT AS IDENTITY` (BIGINT) — `SERIAL` 미사용 |
| Numeric 정밀도 | 통화 `NUMERIC(15,2)`, 환율 `NUMERIC(12,6)` |
| 텍스트 | 짧은 코드 `VARCHAR(N)`, 일반 텍스트 `TEXT`, 다국어 본문 `TEXT` |
| JSON | `JSONB` (확장 속성/MARC 서브필드 등) — `JSON`(인덱싱 불가) 미사용 |
| 한국어 검색 | `pg_trgm` + `unaccent` + 별도 `tsvector` 컬럼 (CAT 도메인) |

### 9.1 PostgreSQL 확장 (Extensions)

| 확장 | 용도 |
|---|---|
| `pgcrypto` | 컬럼 암호화 (회원 PII) |
| `pg_trgm` | 한글 부분일치 검색 (서지 제목·저자) |
| `unaccent` | 발음구별기호 무시 검색 (외국어 서지) |
| `pg_stat_statements` | 쿼리 성능 모니터링 |
| `pgaudit` | 감사 로그 (개인정보 접근) |
| `uuid-ossp` 또는 ULID 함수 | public_id 생성 |
| `btree_gin` | GIN + B-Tree 복합 검색 |

## 10. KORMARC 모델링 원칙 (핵심 설계 결정)

도서관 시스템의 핵심이므로 별도 절로 정리한다.

### 10.1 채택 모델: **하이브리드 (Normalized Field + JSONB Subfield)**

KORMARC는 가변·반복·서브필드를 가지므로 다음 3가지 옵션을 검토하였다.

| 옵션 | 장점 | 단점 | 채택 |
|---|---|---|---|
| A. Full-Normalized (Field/Subfield 행) | 표준화·서브필드 직접 인덱싱 | 1서지 = 수십~수백 행, 조인 비용 큼 | 부분 채택 |
| B. Full-JSONB (서지 1행 + JSONB) | 단순·빠른 read | 인덱싱·검증 어려움 | 미채택 (Main) |
| C. **하이브리드** — Bibliography(요약) + MarcField(N행) + Subfield(JSONB) | 검색성 + 유연성 | 모델 약간 복잡 | **채택** |

### 10.2 채택 모델 상세

```
tlp_cat_bibliography (서지 1행, 요약 컬럼: title, author, isbn, pub_year, language, ...)
   └─ tlp_cat_marc_field (서지N: 1서지 = 다중 필드 행)
        - tag VARCHAR(3)              -- 020, 245, 700 ...
        - ind1 CHAR(1), ind2 CHAR(1)
        - occurrence_no SMALLINT      -- 반복 필드 순번
        - subfields JSONB             -- [{"code":"a","value":"..."}]
        - control_value TEXT          -- 제어필드(00X)일 경우 직접 값
        - field_type CHAR(1)          -- 'C'(Control) / 'D'(Data)
```

### 10.3 설계 결정 이유

1. **검색은 요약 컬럼 + GIN 인덱싱(tsvector)** 으로 처리 — OPAC 검색 성능 확보. (서지 5천만건에서 N서브필드 조인은 불가능)
2. **편목 편집은 marc_field 단위 행 + subfields JSONB** — UI에서 필드/지시기호/서브필드 단위 편집 자연스러움.
3. **반복 필드 관리** — `occurrence_no`로 순번 보장 (700$a 반복 저자).
4. **서지 1건 저장 ≤ 1초** (CAT 비기능 요구사항) — 요약 컬럼 갱신은 트리거/애플리케이션에서 일관성 유지.
5. **JSONB 서브필드 인덱싱** — 필요 시 `(tag, subfields->>'a')` 표현식 인덱스 또는 GIN(`subfields jsonb_path_ops`).
6. **MARC21 변환** — 동일 모델로 표현 가능 (tag·ind·subfields 동일 구조).

### 10.4 권위/분류/주제 분리

- 권위레코드(`tlp_cat_authority`)는 별도 엔티티 — 100/600/650/700 등에서 `authority_id` 참조.
- 분류표(`tlp_cmn_classification`)는 코드 마스터 — 056/082/090에서 코드값 참조.

### 10.5 외부 표준 import/export

- 원본 MARC 텍스트(.mrc, MARCXML, MARC-in-JSON) 보존: `tlp_cat_marc_record_raw` (BLOB/TEXT) — import/audit 용.
- 파싱 후 정규화 데이터 모델에 적재.

## 11. 등록번호(Accession Number) 채번 정책

장서관리(COL) 핵심 결정.

### 11.1 채번 규칙 마스터 (`tlp_col_accession_rule`)

| 필드 | 설명 |
|---|---|
| library_id | 관 단위 분기 (COL-002) |
| item_type | 자료유형별 분기 (단행본/연속간행물/비도서) |
| prefix | 접두어 (예: `M`, `J`, `D`) |
| year_part | `YYYY`(4) / `YY`(2) / 없음 |
| seq_width | 일련번호 자릿수 |
| current_seq | 현재 시퀀스 값 (FOR UPDATE 잠금) |

### 11.2 채번 트랜잭션

- 단일 트랜잭션 내 `SELECT ... FOR UPDATE`로 잠금 후 `current_seq + 1` 후 조립.
- 또는 PostgreSQL `nextval()` 시퀀스를 관·유형 조합으로 동적 관리 (대량 일괄 등록 시 유리).
- 유일성: `uk_copy_accession (tenant_id, library_id, accession_no)`.
